## Core Concepts

A product demo is the moment where interest becomes conviction. Presentation frameworks like reveal.js demonstrate that modern presentations are built with web technologies (HTML, CSS, JavaScript), enabling interactive, responsive, and embeddable slide experiences. But the technology is secondary to the structure and delivery.

### Demo Structure: The Problem-Solution Arc

Every effective demo follows this arc:

1. **Set the context (30 seconds):** Restate the prospect's problem in their words. "You mentioned that tracking deals across spreadsheets costs your team 5 hours per week."
2. **Show the outcome (60 seconds):** Demo the end result first, not the setup. Show the dashboard, the report, the completed workflow. Let them see the destination before the journey.
3. **Walk the path (3-5 minutes):** Now show how to get there, step by step. Only show features that map to their stated needs.
4. **Handle the "what about" (2-3 minutes):** Address objections with live interaction. "What if a deal has multiple stakeholders?" Show it.
5. **Close with next steps (30 seconds):** End with a clear ask, not a whimper.

### Slide Design Principles

When using slides (reveal.js or any tool), follow these rules:

- **One idea per slide.** If a slide has more than one concept, split it.
- **Visual over text.** A screenshot, diagram, or number beats a paragraph every time.
- **Nested slides for depth.** reveal.js supports vertical (nested) slides. Use them for detail that only some audiences need, keeping the main flow clean.
- **Code highlighting for technical audiences.** Show real code snippets with syntax highlighting when demoing to developers.
- **Progressive disclosure.** Reveal bullet points one at a time using fragments to control attention.

### Demo Anti-Patterns

| Anti-Pattern | Why It Fails | Fix |
|-------------|-------------|-----|
| Feature tour | No connection to buyer's problem | Map every feature shown to a stated need |
| Happy path only | Feels rehearsed and fragile | Show one "what if" scenario to build trust |
| Too long | Attention drops after 10 minutes | Keep core demo under 10 minutes |
| No interaction | Prospect zones out | Ask questions: "Is this how your team would use it?" |
| Starting with login | Boring, wastes time | Start at the value moment, skip setup |

### Interactive Demo Tools

Modern sales teams use interactive demo tools that let prospects click through a product without needing a live environment. Building these with web technologies:

- **reveal.js** for slide-based presentations with code examples, auto-animate transitions, and speaker notes.
- **Embedded iframes** for live product walkthroughs within slides.
- **PDF export** for leave-behind materials after the demo.

### Pre-Demo Checklist

- [ ] Confirmed prospect's top 3 pain points
- [ ] Customized demo environment with relevant data
- [ ] Tested all demo flows end-to-end
- [ ] Prepared for 3 likely objections
- [ ] Backup plan if live demo fails (screenshots, recorded video)

## Key Takeaways

- Show the outcome first, then walk the path; never start with a feature tour.
- Keep demos under 10 minutes and map every feature shown to a stated buyer need.
- Use one idea per slide and visual evidence over text walls.
- Build interaction into demos by asking questions and handling live "what about" scenarios.
- Always have a backup plan if the live environment fails.
