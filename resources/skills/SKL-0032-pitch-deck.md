# Pitch Deck

Create a structured pitch deck outline for investors, stakeholders, or partners. Natural output after PRD and Problem Stress Test.

## 12-Slide Structure

| Slide | Content |
|-------|---------|
| 1. Title | Product name, tagline, founder name |
| 2. Problem | Problem in user language, scale, current painful workarounds |
| 3. Solution | One-sentence solution, 3 key features, demo placeholder |
| 4. Market Size | TAM, SAM, SOM with data sources |
| 5. Business Model | How you make money, pricing, unit economics |
| 6. Traction | What you have built, users/revenue, validation signals |
| 7. Competition | Landscape matrix, primary differentiator |
| 8. Go-to-Market | First 100 users strategy, growth approach, distribution advantage |
| 9. Team | Founder background, key hires needed, advisors |
| 10. Financials | Revenue projections, key assumptions, burn rate |
| 11. The Ask | Raise amount, use of funds, milestones funded |
| 12. Closing | Strong closing statement (not "thank you"), contact info |

## Procedure

### 1. Gather Context

Read available documents: PRD or GDD, Problem Stress Test, Competitor Analysis, and key decisions. If minimal context exists, interview the user.

### 2. Build Each Slide

Write content for all 12 slides following the structure above. Pull from validated sources where possible.

### 3. Apply Pitch Principles

- First and last slides get the most screen time. Make them count.
- Lead with the problem narrative, not the solution description.
- Drive emotional engagement: investors should envision using the product.
- The deck must survive brutal Q&A.

## Output

Save to `docs/PITCH_DECK.md` as a content outline (not actual slides). Each section maps to one slide for the user's presentation tool.

## Key Rules

- Produces an outline document, not actual slide files
- Financial projections are frameworks with clearly labeled assumptions
- Never fabricate traction or metrics. If pre-launch, say so honestly.
- Market size estimates flag rough numbers explicitly.
