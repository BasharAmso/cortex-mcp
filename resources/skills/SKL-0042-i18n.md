# Internationalization (i18n)

Internationalization is structuring your app so it can be adapted to different languages and regions without code changes. Localization (l10n) is the actual translation and formatting for a specific locale.

## Library Comparison

| Library | Framework | Bundle Size | Features | Best For |
|---------|-----------|-------------|----------|----------|
| next-intl | Next.js | ~5 KB | Server components, type-safe, routing | Next.js apps (recommended) |
| react-intl (FormatJS) | React | ~12 KB | ICU message format, rich formatting | Apps needing ICU standard compliance |
| i18next + react-i18next | Any | ~15 KB | Plugins ecosystem, namespaces | Framework-agnostic, large apps |
| Paraglide (inlang) | Any | ~0 KB (compiled) | Treeshaken, type-safe, compiled | Maximum performance |
| vue-i18n | Vue | ~10 KB | Vue-native, SFC support | Vue apps |

## Translation File Structure

Organize translations by locale and namespace:

```
locales/
  en/
    common.json      # Shared strings (nav, buttons, errors)
    auth.json         # Login, signup, password reset
    dashboard.json    # Dashboard-specific strings
  ar/
    common.json
    auth.json
    dashboard.json
```

### Key Naming Conventions
- Use dot-notation keys: `"auth.login.title"` not `"loginPageTitle"`
- Group by feature, not by component
- Keep keys descriptive: `"cart.empty.message"` not `"msg1"`
- Never use the English text as the key (it will break when the English copy changes)

## Locale Routing

Two main approaches:

| Approach | URL Example | Pros | Cons |
|----------|-------------|------|------|
| Path prefix | `/ar/about` | SEO-friendly, shareable | Requires routing config |
| Cookie/header | `/about` + `Accept-Language` | Simpler URLs | Bad for SEO, not shareable |

Path prefix is strongly recommended for public-facing sites. Use the user's browser language for initial redirect, then let them override.

## Date, Number, and Currency Formatting

Never format these manually. Use `Intl` APIs built into every modern browser:

- **Dates:** `new Intl.DateTimeFormat('ar-EG', { dateStyle: 'long' })`
- **Numbers:** `new Intl.NumberFormat('de-DE', { style: 'decimal' })`
- **Currency:** `new Intl.NumberFormat('ja-JP', { style: 'currency', currency: 'JPY' })`
- **Relative time:** `new Intl.RelativeTimeFormat('en', { numeric: 'auto' })`

These handle locale-specific decimal separators, date ordering, and currency symbol placement automatically.

## RTL (Right-to-Left) Support

Languages like Arabic, Hebrew, and Persian read right-to-left. This affects the entire layout.

### Implementation Checklist
- Set `dir="rtl"` on `<html>` when locale is RTL
- Use CSS logical properties: `margin-inline-start` not `margin-left`, `padding-inline-end` not `padding-right`
- Flexbox and Grid already respect `dir` for ordering
- Mirror icons that imply direction (arrows, progress indicators). Do NOT mirror: logos, clocks, checkmarks, media controls
- Test with actual RTL content, not mirrored English

## Common Mistakes

- **Concatenating translated strings.** `"Hello " + name + ", welcome"` breaks in languages with different word order. Use interpolation: `"greeting": "Hello {name}, welcome"`.
- **Hardcoding plurals.** English has 2 plural forms. Arabic has 6. Use ICU plural syntax or your library's plural handling.
- **Forgetting text expansion.** German text is 30-40% longer than English. UI must handle varying text lengths without breaking layout.
- **Translating developer-facing strings.** Error codes, log messages, and API responses stay in English. Only translate user-facing content.
- **No context for translators.** Add comments or descriptions to translation keys so translators understand where the string appears.
