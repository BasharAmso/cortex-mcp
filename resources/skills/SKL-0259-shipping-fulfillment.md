## Core Concepts

Shipping connects the digital order to a physical delivery. The fulfillment system must calculate rates, restrict options by geography, manage the pick-pack-ship workflow, and keep customers informed with tracking updates.

### Fulfillment Data Model

```
Fulfillment Provider (e.g., "manual", "fedex", "shippo")
└── Shipping Profile (groups products with similar shipping rules)
    └── Service Zone (geographic boundary)
        └── Shipping Option (method + pricing)
            ├── Flat rate: $5.99
            ├── Weight-based: $0.50/lb
            └── Carrier-calculated: live rate from API
```

### Shipping Option Types

| Type | How It Works | Best For |
|------|-------------|----------|
| **Flat rate** | Fixed price regardless of cart contents | Simple stores, subscription boxes |
| **Weight-based** | Price tiers by total weight | Physical goods with varied sizes |
| **Price-based** | Free shipping above threshold (e.g., free over $50) | Conversion optimization |
| **Carrier-calculated** | Real-time rate from carrier API | Accuracy, heavy/oversized items |

### Geographic Restrictions with Service Zones

Shipping options are scoped to service zones defined by geographic boundaries:

- **Country-level:** Ship to US, CA, UK
- **State/province-level:** Different rates for Alaska and Hawaii
- **Postal code ranges:** Same-day delivery in select ZIP codes

Each zone links to one or more shipping options. If a customer's address falls outside all zones, no shipping options appear and checkout cannot proceed. This prevents orders you cannot fulfill.

### Fulfillment Workflow

```
Order Placed → Awaiting Fulfillment → Picked → Packed → Shipped → Delivered
                                                  ↓
                                            Label Generated
                                            Tracking Number Assigned
                                            Customer Notified
```

Each fulfillment record contains:

| Field | Purpose |
|-------|---------|
| `location_id` | Warehouse or store fulfilling the order |
| `provider_id` | Which carrier or fulfillment service |
| `items[]` | Line items with SKU, quantity, barcode |
| `labels[]` | Shipping labels generated by the provider |
| `tracking_number` | Carrier tracking identifier |
| `delivery_address` | Destination with country, city, postal code |

### Split Shipments

When items ship from different warehouses or at different times, create multiple fulfillment records for one order. Each fulfillment has its own tracking number and status. The order status reflects the aggregate: it is not "fulfilled" until all fulfillments are complete.

### Returns and Exchanges

| Step | Action |
|------|--------|
| **Request** | Customer initiates return with reason and item selection |
| **Approve** | Staff reviews and approves (or auto-approve by policy) |
| **Ship back** | Generate return label, provide tracking |
| **Receive** | Warehouse confirms receipt and inspects items |
| **Resolve** | Refund, exchange, or store credit |

Track return fulfillments separately from outbound fulfillments. Link both to the original order for a complete audit trail.

### Fulfillment Providers

Medusa treats fulfillment providers as pluggable modules. A provider implements a standard interface:

- `calculatePrice()` — return shipping cost for given items and destination
- `createFulfillment()` — generate label and tracking number
- `cancelFulfillment()` — void a shipment before pickup
- `getFulfillmentDocuments()` — return labels, customs forms

This lets you swap carriers or add new ones without changing order or checkout logic.

## Key Takeaways

- Scope shipping options to geographic service zones to prevent unfulfillable orders
- Support multiple fulfillment records per order for split shipments from different warehouses
- Use pluggable fulfillment providers so carrier changes do not affect checkout logic
- Track returns as separate fulfillment records linked to the original order
- Offer free shipping thresholds as a conversion optimization tool
