# Internationalization (i18n)

Internationalization is structuring your app so it can adapt to different languages and regions without code changes. Localization (l10n) is the actual translation and formatting for a specific locale. Following nodebestpractices' principle of separating configuration from code, locale data should be externalized into translation files, never hardcoded in templates.

## Library Comparison

| Library | Framework | Bundle Size | Best For |
|---------|-----------|-------------|----------|
| next-intl | Next.js | ~5 KB | Next.js apps (server components, type-safe, routing) |
| react-intl (FormatJS) | React | ~12 KB | ICU standard compliance, rich formatting |
| i18next + react-i18next | Any | ~15 KB | Framework-agnostic, large apps, plugin ecosystem |
| Paraglide (inlang) | Any | ~0 KB (compiled) | Maximum performance (treeshaken, compiled) |
| vue-i18n | Vue | ~10 KB | Vue apps with SFC support |

## Translation File Structure

Organize translations by locale and namespace:

```
locales/
  en/
    common.json      # Shared strings (nav, buttons, errors)
    auth.json         # Login, signup, password reset
    dashboard.json    # Feature-specific strings
  ar/
    common.json
    auth.json
    dashboard.json
```

### Key Naming Conventions
- Dot-notation keys: `"auth.login.title"` not `"loginPageTitle"`
- Group by feature, not by component
- Keep keys descriptive: `"cart.empty.message"` not `"msg1"`
- Never use English text as the key (breaks when English copy changes)

## Locale Routing

| Approach | URL Example | Pros | Cons |
|----------|-------------|------|------|
| Path prefix | `/ar/about` | SEO-friendly, shareable | Requires routing config |
| Cookie/header | `/about` + `Accept-Language` | Simpler URLs | Bad for SEO, not shareable |

Path prefix is strongly recommended for public-facing sites. Use the browser language for initial redirect, then let the user override.

## Date, Number, and Currency Formatting

Never format these manually. Use `Intl` APIs built into every modern browser and Node.js:

- **Dates:** `new Intl.DateTimeFormat('ar-EG', { dateStyle: 'long' })`
- **Numbers:** `new Intl.NumberFormat('de-DE', { style: 'decimal' })`
- **Currency:** `new Intl.NumberFormat('ja-JP', { style: 'currency', currency: 'JPY' })`
- **Relative time:** `new Intl.RelativeTimeFormat('en', { numeric: 'auto' })`

These handle locale-specific separators, date ordering, and currency symbol placement automatically. Per nodebestpractices' validation principles, validate that locale strings are valid before passing to `Intl`.

## RTL (Right-to-Left) Support

Languages like Arabic, Hebrew, and Persian read right-to-left, affecting entire layout.

### Implementation Checklist
- Set `dir="rtl"` on `<html>` when locale is RTL
- Use CSS logical properties: `margin-inline-start` not `margin-left`
- Flexbox and Grid already respect `dir` for ordering
- Mirror directional icons (arrows, progress). Do NOT mirror: logos, clocks, checkmarks, media controls
- Test with actual RTL content, not mirrored English

## Common Mistakes

- **Concatenating translated strings.** `"Hello " + name` breaks in languages with different word order. Use interpolation: `"greeting": "Hello {name}"`.
- **Hardcoding plurals.** English has 2 plural forms. Arabic has 6. Use ICU plural syntax or your library's plural handling.
- **Forgetting text expansion.** German text is 30-40% longer than English. UI must handle varying lengths.
- **Translating developer-facing strings.** Error codes, logs, and API responses stay in English. Only translate user-facing content.
- **No context for translators.** Add descriptions to keys so translators know where strings appear.
